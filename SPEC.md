## 1. 项目概述

### 1.1 项目背景

FlingTrainer-Manager 是一款基于 Electron 的桌面应用程序，旨在帮助用户下载和管理单机游戏辅助工具（如"风灵月影"修改器），提升玩家在单机游戏中使用训练器的效率与便捷性。

### 1.2 目标用户

- 单机游戏玩家
- 需要频繁获取和使用游戏修改器的用户
- 希望集中管理单机游戏修改器的玩家

### 1.3 核心价值

解决传统方式下手动搜索、下载、管理单机游戏辅助工具繁琐且来源分散的问题，提供集中化、自动化的一站式管理方案。

### 1.4 产品定位

Windows桌面应用，通过网页爬虫技术自动抓取和解析游戏修改器资源，实现一键下载和本地管理。

- 首页加载时间 < 3 秒（首次）
- 搜索结果返回 < 5 秒
- 下载信息获取 < 3 秒
- 界面操作响应 < 100 毫秒

**资源占用**:
- 内存占用 < 500MB（正常使用）
- CPU 占用 < 10%（空闲状态）
- 安装包体积 < 100MB

**并发处理**:
- 支持同时下载多个文件
- 异步处理不阻塞 UI

### 3.2 可靠性要求

**可用性**:
- 应用崩溃率 < 0.1%
- 关键功能可用性 > 99%

**容错性**:
- 网络错误时自动降级使用缓存
- 文件操作失败时提供友好提示
- 异常捕获不导致应用崩溃

**数据完整性**:
- 下载文件校验完整性
- 配置文件损坏时自动重置
- 缓存数据定期清理过期内容

- 使用 ASAR 打包保护源码
- 预加载脚本隔离上下文
- 禁用 Node.js 集成（nodeIntegration: false）

**数据安全**:
- 不存储敏感用户信息

**网络安全**:

### 4.3 数据流

#### 4.3.1 游戏列表加载流程

```
用户点击"所有游戏"
    ↓
渲染进程 → loadGames()
    ↓
预加载脚本 → ipcRenderer.invoke('load-games-from-web')
    ↓
主进程 IPC → fetchHomepageGames()
    ↓
爬虫服务 → axios.get(目标网站)
    ↓
cheerio 解析 → 提取游戏条目
    ↓
缓存检查 → 写入/读取缓存
    ↓
返回数据 → 渲染进程
    ↓
DOM 渲染 → 游戏列表展示
```

#### 4.3.2 下载流程

```
用户点击"下载"按钮
    ↓
检查下载文件夹设置
    ↓
渲染进程 → getDownloadInfo(downloadPageUrl)
    ↓
主进程 → 访问下载页面
    ↓
cheerio 解析 → 提取下载链接、游戏名、版本等
    ↓
返回下载信息
    ↓
渲染进程 → addDownloadTask(taskInfo)
    ↓
主进程 → 添加到下载队列
    ↓
Toast 提示（已添加到下载队列）
    ↓
用户可在"我的修改器 - 下载列表"中查看进度
```

**按钮状态说明**:
- 首页、搜索页面的下载按钮点击后仅显示 Toast 提示，不变更按钮状态
- 下载任务的实际进度和状态在"我的修改器 - 下载列表"页签中统一展示
- 简化前端交互逻辑，降低代码复杂度

## 5. 功能详细说明

### 5.1 页面结构

#### 5.1.1 首页 (home)

**组成元素**:
- 欢迎内容区域
- 搜索栏（输入框 + 搜索按钮 + 清除按钮）
- 搜索结果容器（隐藏/显示）

**交互逻辑**:
1. 默认显示欢迎内容
2. 输入关键词后点击搜索
3. 显示搜索结果区域
4. 点击清除按钮返回初始状态

#### 5.1.2 所有游戏 (all-games)

**组成元素**:
- 页面标题
- 游戏列表容器（网格布局）

**游戏卡片结构**:
- 游戏封面图（200x200）
- 游戏名称
- 操作按钮：
  - "查看详情" → 打开详情页
  - "下载" → 添加到下载队列（显示 Toast 提示，不变更按钮状态）

**交互说明**:
- 点击下载按钮后，用户可在"我的修改器 - 下载列表"中查看实时进度
- 简化首页和搜索页的交互逻辑，避免频繁的状态更新

#### 5.1.3 我的修改器 (cheats)

**组成元素**:
- 页签切换：已下载 / 下载列表
- 文件列表容器

**已下载列表结构**:
- 游戏图标
- 文件信息（名称、类型、大小、日期）
- 操作按钮：
  - 可执行文件："启动"
  - 压缩包："打开文件夹" + 解压提示

**下载列表结构**:
- 统计信息栏：显示总任务数、下载中、排队中、已完成数量
- 移除全部记录按钮：点击后弹出二次确认，清除所有非下载中状态的任务记录
- 任务列表项：
  - 任务名称和状态标签
  - 进度条和详细信息（进度、大小、速度、时间）
  - 操作按钮：
    - 下载中/排队中："取消下载"
    - 已完成："打开文件位置" + "移除记录"
    - 失败/取消："重新下载" + "移除记录"

**排序规则**:
- 默认按任务开始时间倒序排列（最新的在最上方）
- 确保用户优先看到最近添加或正在下载的任务
- 无开始时间的任务自动排在后面

**交互逻辑**:
1. 点击下载按钮后，任务出现在"下载列表"页签
2. 实时显示下载进度、速度等信息
3. 单个任务管理：
   - 下载中/排队中的任务可取消
   - 已完成的任务可打开文件位置或单独移除记录
   - 失败/取消的任务可重新下载或移除记录
4. 批量管理：
   - 点击"移除全部下载记录"按钮
   - 弹出确认对话框："该操作将会移除所有下载记录，不影响实际已下载的工具"
   - 确认后清除所有已完成、失败、取消的任务记录
   - 保留正在下载和排队中的任务

#### 5.1.4 设置 (settings)

**组成元素**:

  - 路径输入框（只读）

3. 点击"打开文件夹"在文件资源管理器中打开下载目录

#### 5.1.5 关于 (about)

**组成元素**:
- 软件介绍

- 作者信息

### 5.2 缓存策略

#### 5.2.1 缓存类型

| 缓存类型 | 路径 | 有效期 | 用途 |
|---------|------|--------|------|
| 游戏列表 | cache/games-cache.json | **12 小时** | 首页和全部游戏列表（包含 base64 图片） |
| 搜索结果 | cache/search-cache/{keyword}.json | **12 小时** | 相同关键词的搜索结果（包含 base64 图片） |
| 下载信息 | cache/download-cache/{urlHash}.json | 10 分钟 | 下载页面详情 |
| 图片缓存 | cache/images/ | 永久（受大小限制） | 游戏封面图本地缓存 |

#### 5.2.2 手动刷新机制

**功能描述**:
- 首页"最近更新"区域右上角提供刷新按钮
- 搜索页面在显示搜索结果后自动显示刷新按钮
- 点击刷新按钮会强制从网络获取最新数据

**交互逻辑**:
1. 用户点击刷新按钮
2. **冷却时间检查**：检查距离上次刷新是否超过 60 秒
   - 如果在冷却期内：提示"请勿频繁刷新，请等待 X 秒后再试"
   - 如果通过检查：继续下一步
3. **每日限额检查**：检查当日剩余刷新次数
   - 如果已达上限（30 次）：提示"今日刷新次数已达上限，请明日再试"
   - 如果有余额：继续下一步并扣减配额
4. 系统跳过缓存检查，直接访问目标网站
5. 比对返回数据与缓存数据的差异
6. 根据比对结果显示提示：
   - **数据没有更新**：新数据与缓存一致
   - **数据已更新**：新数据与缓存不一致，并更新本地缓存
7. 按钮显示旋转动画作为加载反馈

**限制策略**:
```
// 配置参数（constants.js）
export const REQUEST_LIMITS = {
  REFRESH_COOLDOWN: 60 * 1000,      // 刷新冷却时间：60 秒
  MAX_REFRESH_PER_DAY: 30,          // 每日最多刷新次数：30 次
  RANDOM_DELAY_MIN: 1000,           // 搜索最小延迟：1 秒
  RANDOM_DELAY_MAX: 3000,           // 搜索最大延迟：3 秒
};
```

**数据比对算法**:
```
function generateDataHash(data) {
  // 拼接所有游戏的关键信息（名称 + 链接）
  const keyString = data
    .map(game => `${game.name}|${game.downloadPageLink}`)
    .join(';');
  
  // 计算简单 hash 值
  let hash = 0;
  for (let i = 0; i < keyString.length; i++) {
    const char = keyString.charCodeAt(i);
    hash = ((hash << 5) - hash) + char;
    hash = hash & hash;
  }
  return hash.toString(16);
}
```

**注意事项**:
- 缓存时间内（12 小时），除非手动点击刷新按钮，否则优先使用缓存数据
- 强制刷新失败时，自动降级使用过期缓存（如果有）
- 刷新按钮带旋转动画，提升用户体验
- **搜索功能不受每日限额限制**，但会添加 1-3 秒随机延迟
- 所有限制旨在防止过度调用目标网站，符合道德爬虫原则

#### 5.2.3 图片缓存机制

**存储策略**:
- **格式**: 图片以 base64 编码形式嵌入到游戏列表缓存 JSON 中
- **本地文件**: 同时保存原始图片文件到 `cache/images/` 目录
- **命名**: 使用 URL 的 Base64 编码作为文件名，避免重复和冲突
- **大小限制**: 图片缓存总大小限制为 **50MB**
- **清理策略**: 当超过大小限制时，自动删除最旧的文件

**下载流程**:
```
async function downloadAndCacheImage(imageUrl) {
  // 1. 检查是否已缓存
  const cached = getCachedImage(imageUrl);
  if (cached) return cached;
  
  // 2. 下载图片
  const response = await axios.get(imageUrl, {
    headers: DEFAULT_HEADERS,
    responseType: 'arraybuffer',
    timeout: 10000
  });

  const imageData = Buffer.from(response.data);

}
```

#### 5.2.4 搜索功能保护机制

**随机延迟策略**:
- **目的**：模拟真实用户行为，避免被识别为自动化脚本
- **延迟范围**：1-3 秒（随机）
- **作用域**：仅针对搜索请求（`search-games`）
- **实现方式**：axios 请求拦截器

**代码示例**:
```
// http.js - 创建客户端时启用随机延迟
const client = createHttpClient({}, true); // enableRandomDelay = true

// 拦截器自动添加延迟
client.interceptors.request.use(async (config) => {
  if (config.url?.includes('flingtrainer.com')) {
    await randomDelay(1000, 3000); // 1-3 秒随机延迟
  }
  return config;
});
```

**用户体验**:
- 搜索操作会有 1-3 秒的等待时间
- 属于正常现象，无需额外提示
- 有效降低目标网站服务器压力
- 符合道德爬虫规范

#### 5.2.5 缓存逻辑

```
// 伪代码示例
function fetchData(forceRefresh = false) {
  // 1. 如果强制刷新，跳过缓存检查
  if (forceRefresh) {
    console.log('强制刷新，跳过缓存检查');
  } else {
    // 2. 检查缓存是否存在且未过期
    if (cacheFile exists && now - mtime < TTL) {
      return cachedData; // 使用缓存
    }
  }
  
  // 3. 发起网络请求
  try {
    const data = await httpRequest();
    writeCache(cacheFile, data); // 写入新缓存
    return data;
  } catch (err) {
    // 4. 网络失败时使用过期缓存（如果有）
    if (cacheFile exists) {
      return oldCachedData;
    }
    throw err;
  }
}
```

**强制刷新参数传递**:
- 前端调用：`window.api.loadGames(true)` 或 `window.api.searchGames(keyword, true)`
- IPC 层：`ipcRenderer.invoke('load-games-from-web', forceRefresh)`
- 主进程：`fetchHomepageGames(forceRefresh)`
- 爬虫服务：根据 `forceRefresh` 参数决定是否跳过缓存检查

## 8. 已知限制与风险

### 8.1 技术限制

1. **网页爬虫依赖**
   - 风险：目标网站结构变更会导致解析失败
   - 缓解：监控网站更新，及时调整选择器
   - 长期方案：考虑官方 API（如果有）

2. **缓存一致性**
   - 风险：缓存数据可能过期或不准确
   - 缓解：设置合理的 TTL，定期清理
   - 用户控制：提供"刷新"按钮强制更新

3. **下载源稳定性**
   - 风险：下载链接失效或服务器不可用
   - 缓解：使用缓存的下载信息
   - 用户控制：提示用户稍后重试

### 8.2 法律合规

1. **版权问题**
   - 说明：游戏修改器可能涉及版权争议
   - 建议：用户在知悉风险下自行使用
   - 免责声明：软件仅提供技术实现

2. **用户使用规范**
   - 建议：仅用于单机游戏学习研究
   - 禁止：网络游戏作弊等违规行为
   - 责任：用户自行承担使用后果

### 8.3 安全风险

1. **文件安全性**
   - 风险：下载的文件可能包含恶意代码
   - 建议：配合杀毒软件使用
   - 缓解：仅从可信源下载

2. **隐私保护**
   - 现状：不收集用户个人信息
   - 本地数据：设置文件和缓存仅存储在本地
   - 网络请求：不上传用户数据
